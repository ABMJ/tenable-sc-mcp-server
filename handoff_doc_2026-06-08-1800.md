**Module Organization Rules:**
- Each module = 200-600 lines maximum (keep manageable)
- Group tools by logical functionality (see TOOLS_ROADMAP.md for complete mapping)
- **All new tools MUST go in appropriate module, NEVER in server.py**
- Admin tools go in `tools/admin/` subdirectory
- Server.py contains ONLY: core MCP setup, generic tools (catalog, request, analyze), cache management

**Module File Pattern:**
```python
# tools/[module_name].py
"""[Module description] for Tenable.sc MCP Server."""

from __future__ import annotations
from typing import Any
from ..convenience_tools import validate_ip, build_filters

def register_tools(mcp):
    """Register [module] tools with the MCP server."""
    
    @mcp.tool()
    def tsc_tool_name(
        required_param: str,
        optional_param: str | None = None,
    ) -> dict[str, Any]:
        """
        [Clear description of what tool does]. Use this when you need to:
        - [Use case 1]
        - [Use case 2]
        
        WHEN TO USE THIS TOOL:
        - User asks "[example query]"
        - User needs [specific capability]
        
        DO NOT USE for [alternative use cases] - use [other_tool] instead.
        
        Token Efficiency: ~[range] tokens (vs ~[raw] raw API)
        Cache TTL: [seconds]s ([minutes] minutes)
        
        Args:
            required_param: [Description]
            optional_param: [Description]
        
        Returns:
            Dict with:
            - ok: True/False
            - [data fields]
        """
        # Import tsc_analyze from server
        from .. import server
        tsc_analyze = server.tsc_analyze
        
        # Implementation here
        pass

# Export for testing
__all__ = ["register_tools"]
```

### ⭐ CRITICAL: Centralized Helper Functions (DRY Principle)

**Rule**: ALL shared functionality MUST be centralized in `convenience_tools.py`, never duplicated across tool modules.

**Centralized Helpers:**
```python
# convenience_tools.py - Single source of truth for shared logic

# Input validation
validate_ip(ip: str) -> tuple[bool, str]

# Filter building
build_filters(**kwargs) -> list[dict[str, Any]]
  - Handles 55+ analysis filters
  - Automatic operator-to-range conversion for ALL scoring filters:
    * asset_criticality (ACR): 0-10 scale
    * vpr_score (VPR): 0-10 scale
    * aes_score (AES): 0-1000 scale
    * cvss_v3_base_score: 0-10 scale
    * cvss_v4_base_score: 0-10 scale
    * base_cvss_score (CVSS v2): 0-10 scale
    * epss_score: 0-1 scale
  - Examples: ">7" → "7.1-10", ">=7" → "7.0-10", "<7" → "0-6.9"

# Name-to-ID resolution
resolve_repository_name(name: str) -> str | None
resolve_asset_group_name(name: str) -> str | None
  - Handles multiple API response formats
  - Returns None if not found (graceful failure)
  - Used by all tools that accept repository/asset group names

# Vulnerability summarization
summarize_vulnerabilities(results: list) -> dict
  - Aggregates vulnerability counts by severity
```

**Anti-Pattern (DO NOT DO):**
```python
# ❌ WRONG: Duplicating resolver logic in tool module
def _resolve_repository_name(name: str):
    # ... duplicate implementation ...

# ✅ CORRECT: Import from centralized location
from ..convenience_tools import resolve_repository_name
```

**Benefits of Centralization:**
- Single source of truth - fix bugs once, applies everywhere
- Consistent behavior across all tools
- Easier to add caching/optimization centrally
- Reduced code duplication (saves 100+ lines per tool)
- Future tools automatically get all helper functionality
- Easier testing and maintenance

**When to Centralize:**
- If 2+ tools need the same logic → move to `convenience_tools.py`
- Name-to-ID resolution (repositories, asset groups, etc.)
- Format conversions (operators to ranges, severity mapping)
- Input validation (IP, CIDR, date formats)
- Response parsing/normalization

**When to Keep Local:**
- Tool-specific logic unique to one tool
- Private helper functions used only within single module
- Complex business logic specific to one domain

### Error Handling
- Always validate IP format/CIDR notation using `validate_ip()`
- Return user-friendly error messages in `{"ok": False, "error": "..."}` format
- Handle missing data gracefully (return empty lists, not errors)
- Never expose internal implementation details in error messages

### Caching Strategy
- Use smart TTLs based on data volatility:
  - Vulnerability data: 180s (3 min) - changes frequently
  - Analysis queries: 120s (2 min) - moderate volatility
  - Static data (software/services): 300s (5 min) - rarely changes
- Remove pagination params from cache keys (don't cache per-page)
- Cache per-component for multi-query tools (Tool 1 pattern)
- Use `tsc_analyze()` for all analysis queries (handles caching automatically)

### Token Optimization
- Apply filters server-side in Tenable.sc API, not client-side in Python
- Use pagination for large datasets (default 50, max 200)
- Truncate long fields (synopsis/solution to 200 chars) for summaries
- Return only requested fields (don't fetch all data if not needed)
- Use summary tools first, full details only when necessary

### Testing Requirements
- Test with cache cold (first query) and warm (repeat query)
- Validate all filters work correctly with real data
- Test pagination edge cases (0 results, 1 result, max results)
- Confirm token savings vs raw API calls
- Add test queries to TEST_PROMPTS.md using visual format
- Validate on real Tenable.sc instance, not mock data

## 🐛 SEVEN CRITICAL BUGS FIXED

### Bug 1: Missing Nested Query Wrapper
**Problem**: API returned HTTP 403 "query required" error  
**Root Cause**: Query passed without nested `{"query": {...}}` structure  
**Fix**: Added proper nesting: `{"type": "vuln", "query": {actual query}, "sourceType": "cumulative"}`  
**Location**: `asset_discovery.py:234, 459`  
**Lesson**: Tenable.sc analysis API requires specific query structure

### Bug 2: Incorrect Asset Filter Format
**Problem**: Asset group filter caused API errors  
**Root Cause**: Used array format `[{"id": 3}]` instead of object  
**Fix**: Changed to `{"id": "3", "name": "Windows Hosts"}` (object with both ID and name)  
**Location**: `asset_discovery.py:210`  
**Lesson**: Ask user to check browser DevTools for working query format

### Bug 3: Wrong ACR Field Name
**Problem**: ACR scores showing as 240, 3581, 2247 instead of 0-10 range  
**Root Cause**: Used `item.get("score")` which returns vulnerability count (0-4000+)  
**Fix**: Changed to `item.get("acrScore")` for actual Asset Criticality Rating  
**Location**: `asset_discovery.py:310`  
**Lesson**: Reference working tools for correct field names

### Bug 4: ACR Operator Conversion Precision
**Problem**: `>7` converted to `8-10`, excluding IPs with ACR 7.1-7.9  
**Root Cause**: Integer rounding instead of decimal precision  
**Fix**: Convert `>7` to `7.1-10` and `>=7` to `7.0-10`  
**Location**: `asset_discovery.py:232-258`  
**Lesson**: Range conversions need decimal precision for accurate filtering

### Bug 5: Wrong Analysis Tool for Reverse Lookup
**Problem**: Reverse lookup showed 1 repository but 0 asset groups  
**Root Cause**: Used `sumip` tool which doesn't return asset group membership  
**Fix**: Use `sumasset` tool specifically for asset group queries  
**Location**: `asset_discovery.py:489`  
**Lesson**: Different analysis tools return different data structures

### Bug 6: Missing Asset Group Filtering
**Problem**: Reverse lookup returned all 46 system asset groups  
**Root Cause**: API returns all groups with count data, including 0-count groups  
**Fix**: Filter results by `total > 0` to show only actual memberships  
**Location**: `asset_discovery.py:585`  
**Lesson**: Tenable.sc APIs sometimes return everything, require post-filtering

### Bug 7: Incorrect OS Field Mapping
**Problem**: Operating system not displaying correctly  
**Root Cause**: Used secondary OS field instead of primary  
**Fix**: Changed to use `operatingSystem` as primary field  
**Location**: `asset_discovery.py:309`  
**Lesson**: Check multiple tools to identify canonical field names

### Pattern 6: ACR Operator to Range Conversion
```python
# Convert comparison operators to range format
if operator == '>':
    acr_value = f"{threshold+0.1:.1f}-10"  # >7 → "7.1-10"
elif operator == '>=':
    acr_value = f"{threshold:.1f}-10"     # >=7 → "7.0-10"
elif operator == '<':
    acr_value = f"0.0-{threshold-0.1:.1f}" # <7 → "0.0-6.9"
elif operator == '<=':
    acr_value = f"0.0-{threshold:.1f}"     # <=7 → "0.0-7.0"

filters.append({
    "filterName": "acrScore",
    "operator": "=",
    "value": acr_value  # Use range string, not operator
})
```

# Filter sumasset results by total > 0

## 🧪 TEST PROMPT STYLE GUIDE

### Standard Format for All Test Prompts

**Use this format for ALL test prompts in TEST_PROMPTS.md:**

```
I am testing [tool_name] to [action]. Please format your response as:

✅/❌ TEST STATUS: [PASS/FAIL]
📊 CACHE: [HIT/MISS]
🔢 TOKENS: [count] tokens used
📝 SUMMARY: [one-liner about cache and token performance]
📦 RESULT: [brief summary of returned data with key metrics]

If failed, provide ERROR DETAILS with enough information for the developer to fix. Do not suggest code or fixes.
```

### Why This Format

1. **Visual Icons** (✅/❌/📊/🔢/📝/📦) - Easy to scan for pass/fail status at a glance
2. **Structured Sections** - Consistent across all tests, machine-parseable
3. **Cache Tracking** - Always report HIT/MISS for performance validation
4. **Token Metrics** - Track token usage per query for efficiency validation
5. **Actionable Errors** - "ERROR DETAILS for developer" ensures useful debugging info
6. **No Solutions** - "Do not suggest code" prevents polluting test output with fixes

### Examples

**Good Test Prompt:**
```
I am testing tsc_list_ips to list all IPs in repository "Default". Please format your response as:

✅/❌ TEST STATUS: [PASS/FAIL]
📊 CACHE: [HIT/MISS]
🔢 TOKENS: [count] tokens used
📝 SUMMARY: [one-liner about cache and token performance]
📦 RESULT: Total IPs: [count], First 5: [list]

If failed, provide ERROR DETAILS with enough information for the developer to fix. Do not suggest code or fixes.
```

**Bad Test Prompt (Old Style - DO NOT USE):**
```
use tenable-sc to list all IPs in repository "Default"
```

### Benefits

- User immediately sees ✅ PASS or ❌ FAIL
- Cache metrics are always reported consistently
- Token usage is tracked per query for optimization validation
- Error details are structured and actionable for developers
- Format is consistent across all 25 tools for easy comparison
- Machine-parseable for automated testing (future work)

### When Writing New Tests

1. Always start with "I am testing [tool_name] to [action]"
2. Always include the 5 icon sections (✅/❌, 📊, 🔢, 📝, 📦)
3. Include expected values in RESULT section for validation
4. Specify cache expectation (HIT/MISS) for repeat tests
5. End with error handling instruction

## 🏛️ ARCHITECTURAL DECISIONS

### Decision 1: Nested Query Structure for Analysis
**Format**: 
```python
{
    "type": "vuln",
    "query": {
        "type": "vuln",
        "tool": "sumip",
        "filters": [...]
    },
    "sourceType": "cumulative"
}
```
**Rationale**: Required by Tenable.sc API for `/analysis` endpoint  
**Impact**: All analysis tools must use this format with `tsc_analyze()`  
**Reference**: API documentation doesn't clearly show this - discovered through testing

### Decision 2: Asset Filter Requires Object Format
**Format**: 
```python
{
    "filterName": "asset",
    "operator": "=",
    "value": {"id": "3", "name": "Windows Hosts"}
}
```
**Rationale**: API rejects array format `[{"id": 3}]` or missing name field  
**Impact**: Asset group queries require both ID and name resolution  
**Reference**: Discovered by inspecting browser DevTools network traffic

### Decision 3: Repository Filter Uses Array Format
**Format**:
```python
{
    "filterName": "repository",
    "operator": "=",
    "value": [{"id": "9"}]
}
```
**Rationale**: API expects array with string ID for repository filters  
**Impact**: Different filter format than asset groups (inconsistent API design)  
**Reference**: Confirmed through successful testing

### Decision 4: Dual-Query Reverse Lookup
**Implementation**: Two separate queries
1. `sumip` with IP filter → returns repository list
2. `sumasset` with IP filter → returns asset group list

**Rationale**: No single analysis tool returns both data types  
**Impact**: Reverse lookup makes 2 API calls (both cached separately)  
**Benefit**: Each query cached independently for better hit rates

### Decision 5: ACR Operators Convert to Ranges
**Implementation**: Convert comparison operators (>, >=, <, <=) to range strings
- `>7` → `"7.1-10"`
- `>=7` → `"7.0-10"`
- `<7` → `"0.0-6.9"`
- `<=7` → `"0.0-7.0"`

**Rationale**: Tenable.sc analysis API expects range format, not operators  
**Impact**: All scoring filters (ACR, VPR, CVSS, EPSS) need conversion  
**Benefit**: Supports intuitive user queries like "ACR > 7"

### Decision 6: MCP Client Tool Selection Guidance
**Implementation**: Enhanced all tool docstrings with:
- "WHEN TO USE THIS TOOL" section with user query examples
- "DO NOT USE" warnings to prevent misuse
- Clear business use cases

**Rationale**: Help MCP clients intelligently choose correct tool for user intent  
**Impact**: Improved user experience, fewer wrong tool selections  
**Benefit**: Reduces confusion between similar tools (e.g., profile vs summary vs full)
