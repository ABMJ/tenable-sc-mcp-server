MCP Client → tsc_request(method="GET", path="/repository") 2. Server checks cache → Cache miss 3. Server → Tenable.sc API 4. API returns data 5. Server caches result (TTL: 30 minutes) 6. Server → MCP Client 7. Next request → Cache hit (2ms response!) ✨ ``` ### Write Operation (POST/PUT/DELETE) ``` 1. MCP Client → tsc_request(method="POST", path="/scan") 2. Server → Tenable.sc API (no cache check) 3. API processes write 4. Server invalidates related cache (scan*) 5. Server → MCP Client ``` ## Cache Architecture ### Cache Key Generation ```python resource[:object_id][:fields=...][:params=hash] Examples: - "catalog" - "repository:5" - "scan:10:fields=id,name" - "analysis:params=a3f8bc9d" ``` ### TTL Hierarchy ``` Static (24h) Semi-Static (30m) Dynamic (5m) Real-time (1m) ├─ catalog ├─ repository ├─ asset ├─ scan ├─ plugin ├─ scanPolicy ├─ assetTemplate ├─ scanResult └─ pluginFamily ├─ credential └─ query └─ analysis ├─ alert └─ user ``` ## Deployment Architectures ### Single Instance (Development) ``` ┌─────────────┐ │ MCP Server │ │ + In-Memory │ │ Cache │ └─────────────┘ ``` ### Production (Single Server) ``` ┌─────────────┐ ┌────────┐ │ MCP Server │────▶│ Redis │ └─────────────┘ └────────┘ ``` ### Multi-Instance (High Availability) ``` ┌─────────────┐ │ MCP Server 1│────┐ └─────────────┘ │ ├───▶ ┌────────┐ ┌─────────────┐ │ │ Redis │ │ MCP Server 2│────┤ │(Shared)│ └─────────────┘ │ └────────┘ │ ┌─────────────┐ │ │ MCP Server 3│────┘ └─────────────┘ Shared cache = Consistent responses ``` ## Component Details ### MCP Server (`server.py`) -
