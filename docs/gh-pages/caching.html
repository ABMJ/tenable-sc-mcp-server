[Troubleshooting](#troubleshooting) --- ## Quick Start ### Default Setup (In-Memory Cache) Caching is **enabled by default** with in-memory storage: ```bash # Just run the server - caching works out of the box! tenable-sc-mcp --transport streamable-http ``` **Benefits**: Zero configuration, instant start, perfect for development and single-instance deployments. ### Production Setup (Redis Cache) For production or multi-instance deployments, use Redis: ```bash # Start Redis docker run -d --name redis -p 6379:6379 redis:7-alpine # Configure cache backend export TSC_CACHE_BACKEND=redis export TSC_CACHE_REDIS_HOST=localhost # Run server tenable-sc-mcp --transport streamable-http ``` **Benefits**: Persistent cache, shared across multiple servers, production-grade reliability. --- ## Cache Backends ### In-Memory Cache **Best for**: - Development and testing - Single-instance deployments - Quick prototyping - Environments without Redis **Characteristics**: - ✅ Zero dependencies - ✅ Ultra-fast (<1ms operations) - ✅ Thread-safe - ❌ Lost on restart - ❌ Not shared between instances **Configuration**: ```bash TSC_CACHE_BACKEND=memory ``` ### Redis Cache **Best for**: - Production deployments - Multi-instance setups - High-availability environments - Long-running servers **Characteristics**: - ✅ Persistent across restarts - ✅ Shared across instances - ✅ Production-tested - ✅ Distributed invalidation - ⚠️ Requires Redis server **Configuration**: ```bash TSC_CACHE_BACKEND=redis TSC_CACHE_REDIS_HOST=localhost TSC_CACHE_REDIS_PORT=6379 TSC_CACHE_REDIS_DB=0 TSC_CACHE_REDIS_PASSWORD=optional-password ``` --- ## Configuration ### Environment Variables All cache settings use the `TSC_CACHE_*` prefix: | Variable | Default | Description | |----------|---------|-------------| | `TSC_CACHE_ENABLED` | `true` | Enable/disable caching | | `TSC_CACHE_BACKEND` | `memory` | Backend type: `memory` or `redis` | | `TSC_CACHE_REDIS_HOST` | `localhost` | Redis server hostname | | `TSC_CACHE_REDIS_PORT` | `6379` | Redis server port | | `TSC_CACHE_REDIS_DB` | `0` | Redis database number | | `TSC_CACHE_REDIS_PASSWORD` | None | Redis password (optional) | ### Example Configurations #### Development (.env.dev) ```bash TSC_CACHE_ENABLED=true TSC_CACHE_BACKEND=memory ``` #### Production (.env.prod) ```bash TSC_CACHE_ENABLED=true TSC_CACHE_BACKEND=redis TSC_CACHE_REDIS_HOST=redis.internal TSC_CACHE_REDIS_PORT=6379 TSC_CACHE_REDIS_PASSWORD=secure-password ``` #### Disable Cache (.env.nocache) ```bash TSC_CACHE_ENABLED=false ``` --- ## TTL Strategy The cache uses **intelligent Time-To-Live (TTL)** based on data volatility: ### Static Data (24 hours) **Never or rarely changes**: - API Catalog - Plugin Information - Plugin Families **Why 24 hours?**: This data is immutable or updated daily by Tenable. **Cache Key Examples**: ``` catalog:compact=true plugin:19506 pluginFamily:Windows ``` ### Semi-Static Data (30 minutes) **Occasional changes**: - Repositories - Scan Policies - Credentials - User/Group Information - Alerts **Why 30 minutes?**: Configuration changes are infrequent but should propagate reasonably fast. **Cache Key Examples**: ``` repository:5 scanPolicy:10 credential:auth-scanner ``` ### Dynamic Data (5 minutes) **Regular changes**: - Assets (IP addresses, hostnames) - Asset Templates - Queries **Why 5 minutes?**: Balances freshness with cache efficiency for data that changes daily/weekly. **Cache Key Examples**: ``` asset:192.168.1.100 assetTemplate:servers ``` ### Real-Time Data (1 minute) **Frequent changes**: - Active Scans - Scan Results - Analysis Queries **Why 1 minute?**: These change frequently but can tolerate brief delays for cache efficiency. **Cache Key Examples**: ``` scan:running-scan-123 scanResult:latest analysis:vuln-query-abc ``` ### TTL Customization Modify `DEFAULT_TTL_SECONDS` in `src/tenable_sc_mcp/cache.py`: ```python DEFAULT_TTL_SECONDS = { "catalog": 86400, # 24 hours "repository": 1800, # 30 minutes <- Adjust as needed "scan": 60, # 1 minute # Add custom resources "myCustomResource": 600, # 10 minutes } ``` --- ## Cache Management ### Check Cache Statistics ```bash # Using MCP tool { "tool": "tsc_cache_stats" } ``` **Example Output**: ```json { "ok": true, "enabled": true, "backend": "redis", "key_count": 1247, "metrics": { "hits": 8420, "misses": 1580, "hit_rate": "84.2%", "total_requests": 10000, "uptime_hours": "12.5" } } ``` ### Clear Cache #### Clear All Cache ```bash { "tool": "tsc_cache_clear" } ``` #### Clear Specific Pattern ```bash # Clear all scan cache { "tool": "tsc_cache_clear", "pattern": "scan" } # Clear specific scan { "tool": "tsc_cache_clear", "pattern": "scan:10" } # Clear all plugins { "tool": "tsc_cache_clear", "pattern": "plugin" } ``` ### Automatic Invalidation Cache is **automatically invalidated** on write operations: ```bash # This invalidates all scan cache entries { "tool": "tsc_request", "method": "POST", "path": "/scan/10/launch" } ``` **Invalidation Rules**: - `POST` / `PUT` / `PATCH` / `DELETE` → Clears related cache - Scan modifications → Clears scan and scanResult cache - Asset changes → Clears asset and analysis cache - Repository changes → Clears repository and scan cache --- ## Performance ### Expected Improvements | Scenario | Without Cache | With Cache | Improvement | |----------|---------------|------------|-------------| | Plugin Lookup | 500-1000ms | 2-5ms | **99%** | | Repository List | 300-500ms | 2-5ms | **99%** | | Asset Query | 200-400ms | 2-10ms | **98%** | | Catalog Access | 100-200ms | 1-2ms | **99%** | | Complex Workflow | 30-60s | 1-5s | **90-95%** | ### Token Usage Reduction | Workflow | Token Reduction | Monthly Savings (50 sessions/day) | |----------|-----------------|-----------------------------------| | Plugin Analysis | 80-90% | $120-160 | | Asset Queries | 60-70% | $80-100 | | Scan Analysis | 70-80% | $100-120 | | **Average** | **70%** | **$180** | ### Real-World Example **Before Caching**: ``` Workflow: "Analyze critical vulnerabilities" - 10 API calls (plugins, scans, assets) - Total time: 5-7 seconds - Tokens used: 50,000 Repeated 5 times = 25-35 seconds, 250,000 tokens ``` **After Caching** (after first run): ``` Workflow: "Analyze critical vulnerabilities" - 10 cache hits - Total time: 20-50ms - Tokens used: 5,000 Repeated 5 times = 100-250ms, 25,000 tokens Savings: 99% time, 90% tokens! ``` ### Monitoring Cache Performance #### Hit Rate Target - **Good**: >60% hit rate - **Excellent**: >80% hit rate - **Outstanding**: >90% hit rate #### What Affects Hit Rate? **Increases hit rate**: - Repeated queries - Read-heavy workloads - Long-running sessions - Multiple users **Decreases hit rate**: - Frequent write operations - Short TTLs - Diverse query patterns - Manual cache clears --- ## Troubleshooting ### Cache Not Working **Symptom**: All requests show as "cache misses" **Solutions**: ```bash # 1. Check cache is enabled tsc_cache_stats() # Should show "enabled": true # 2. Check backend configuration echo $TSC_CACHE_BACKEND # Should be "memory" or "redis" # 3. Verify Redis connection (if using Redis) redis-cli ping # Should return "PONG" # 4. Check server logs docker logs tenable-sc-mcp # Look for "Cache initialized" message ``` ### Redis Connection Failed **Symptom**: Server falls back to memory cache **Solutions**: ```bash # 1. Verify Redis is running docker ps | grep redis # 2. Check Redis connectivity redis-cli -h $TSC_CACHE_REDIS_HOST ping # 3. Check network connectivity telnet $TSC_CACHE_REDIS_HOST 6379 # 4. Verify Redis password (if set) redis-cli -h $TSC_CACHE_REDIS_HOST -a $TSC_CACHE_REDIS_PASSWORD ping ``` ### Low Hit Rate **Symptom**: Hit rate <40% **Solutions**: ```bash # 1. Check TTL settings # Increase TTLs for stable data in cache.py # 2. Analyze query patterns tsc_cache_stats() # Look at hits vs misses # 3. Reduce manual cache clears # Only clear when necessary # 4. Pre-warm cache on startup # (Coming in future release) ``` ### High Memory Usage **Symptom**: Server using too much RAM **Solutions**: ```bash # 1. Switch to Redis TSC_CACHE_BACKEND=redis # 2. Reduce TTLs # Modify DEFAULT_TTL_SECONDS in cache.py # 3. Implement cache size limits # (Coming in future release) # 4. Clear old cache entries tsc_cache_clear() ``` ### Stale Data **Symptom**: Cache returns outdated information **Solutions**: ```bash # 1. Clear specific cache tsc_cache_clear(pattern="resource") # 2. Reduce TTL for that resource type # Edit DEFAULT_TTL_SECONDS in cache.py # 3. Force cache invalidation after writes # Already automatic for write operations # 4. Disable cache for real-time data TSC_CACHE_ENABLED=false ``` --- ## Best Practices ### 1. Use Redis in Production ```bash # Production configuration TSC_CACHE_BACKEND=redis TSC_CACHE_REDIS_HOST=redis.internal ``` ### 2. Monitor Cache Performance ```bash # Check metrics regularly tsc_cache_stats() # Target: >60% hit rate ``` ### 3. Tune TTLs for Your Use Case ```python # For very stable environments DEFAULT_TTL_SECONDS = { "repository": 3600, # 1 hour instead of 30 min } # For highly dynamic environments DEFAULT_TTL_SECONDS = { "asset": 60, # 1 minute instead of 5 min } ``` ### 4. Clear Cache After Major Changes ```bash # After bulk imports or major config changes tsc_cache_clear() ``` ### 5. Use Patterns for Selective Clearing ```bash # Only clear what changed tsc_cache_clear(pattern="scan") # After scan policy changes tsc_cache_clear(pattern="asset") # After asset import ``` --- ## Advanced Topics ### Cache Key Structure ``` resource[:object_id][:fields=...][:params=hash] Examples: repository repository:5 scan:10:fields=id,name analysis:params=a3f8bc9d ``` ### Thread Safety All cache operations are **thread-safe**: - Uses `threading.RLock` for InMemoryCache - Redis is inherently thread-safe ### Distributed Caching With Redis: - Multiple MCP instances share cache - Invalidation propagates automatically - No cache synchronization needed --- ## Migration Guide ### From v0.1.0 to v0.2.0 Caching is **automatically enabled** - no migration needed! **Optional**: Switch to Redis for production: ```bash # 1. Start Redis docker run -d --name redis -p 6379:6379 redis:7-alpine # 2. Update configuration echo "TSC_CACHE_BACKEND=redis" >> ~/.tenable-sc-mcp.env # 3. Restart server docker compose restart ``` --- ## FAQ **Q: Is caching secure?** A: Yes. Cache only stores data your API key can access. No credential caching. **Q: Can I disable caching?** A: Yes. Set `TSC_CACHE_ENABLED=false`. **Q: Does caching work with multiple Tenable.sc instances?** A: Yes. Each instance can have its own cache prefix. **Q: What happens if Redis crashes?** A: Server falls back to in-memory cache automatically. **Q: Can I see what's in the cache?** A: Use `tsc_cache_stats()` for metrics. Direct inspection coming in future release. **Q: Does cache persist across server restarts?** A: Yes with Redis, no with in-memory cache. --- ## Resources - [Architecture Diagram](architecture.html) - [Test Plan](../TEST_PLAN.md) - [Performance Benchmarks](performance.html) - [GitHub Repository](https://github.com/ABMJ/tenable-sc-mcp-server) --- **Last Updated**: June 5, 2026 **Applies To**: v0.2.0-dev and later
