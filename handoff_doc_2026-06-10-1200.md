## 🏗️ NEW DESIGN PRINCIPLE: Self-Documenting Filter System

### Problem Statement

When MCP clients (OpenCode/Claude) tried to use tools with filters, they encountered:
1. **Parameter name confusion**: Used `acr_score` instead of `asset_criticality`
2. **Format errors**: Used `">=7"` instead of `"7-10"` range format
3. **Silent failures**: Filters ignored without warning or error
4. **No documentation reference**: LLMs had to guess parameter names

**Root Cause**: 55+ filter parameters with no machine-readable documentation exposed via MCP protocol.

### Solution Architecture

**Multi-Tier Documentation System:**

```
┌──────────────────────────────────────────────────────────────┐
│ COMMON_FILTERS Dict (convenience_tools.py)                   │
│ - Single source of truth for all 55+ filters                │
│ - Maps friendly names → Tenable.sc API names                │
└──────────────────────────────────────────────────────────────┘
                           │
                    ┌──────┴──────┐
                    │              │
                    ▼              ▼
┌─────────────────────────────┐   ┌──────────────────────────┐
│ MCP Resource (NEW)          │   │ build_filters() (ENHANCED)│
│ tenable-sc://filters/ref    │   │ - Validates parameters    │
│ - Auto-generated docs       │   │ - Logs warnings           │
│ - All 55+ filters           │   │ - Helpful error messages  │
│ - Examples & troubleshooting│   └──────────────────────────┘
└─────────────────────────────┘
                │
                ▼
┌─────────────────────────────────────────────────────────────┐
│ MCP Client / LLM                                            │
│ - Fetches filter docs at startup or on-demand              │
│ - Uses correct parameter names                              │
│ - Applies correct format (ranges vs operators)              │
│ - Self-educates instead of guessing                         │
└─────────────────────────────────────────────────────────────┘
```

### Implementation Components

**1. MCP Resource (tenable-sc://filters/reference)**

Location: `src/tenable_sc_mcp/resources/filter_reference.py`

Features:
- Auto-generated from `COMMON_FILTERS` dict (never out of sync)
- Grouped by category (Asset, Vulnerability, Scoring, etc.)
- Includes format requirements (range vs exact)
- 15+ practical examples
- Troubleshooting section with common mistakes
- 10,000+ words of comprehensive documentation

**2. Filter Validation (build_filters enhancement)**

Location: `src/tenable_sc_mcp/convenience_tools.py` lines 282-355

New Features:
- `validate` parameter (default: True)
- Tracks unknown parameters
- Logs WARNING with helpful guidance
- Points to MCP resource for valid filters
- Includes common mistake hints

Example Warning:
```
WARNING: Unknown filter parameters will be ignored: acr_score, hostname.
For valid filter names, see MCP resource: tenable-sc://filters/reference
Common mistakes: 'acr_score' should be 'asset_criticality', 
'hostname' should be 'dns_name'.
```

**3. Enhanced Tool Docstrings**

Pattern for all filter-supporting tools:
```python
"""
[Tool description...]

FILTERS:
This tool supports 55+ Tenable.sc analysis filters via **kwargs.
For COMPLETE filter reference, fetch MCP resource: tenable-sc://filters/reference

COMMON FILTERS (Quick Reference):

Scoring Filters (RANGE FORMAT "min-max" REQUIRED):
    asset_criticality="7-10"      # ACR range (0-10) - DO NOT use ">7"
    vpr_score="8-10"               # VPR range (0-10)
    aes_score="600-1000"           # AES range (0-1000)
    [...]

[...]
```

### Benefits

**For LLMs/MCP Clients:**
- Self-documenting API via MCP resource
- Correct parameter names without guessing
- Proper format usage (ranges vs operators)
- Comprehensive examples for complex queries

**For Users:**
- Filters work as expected
- Clear warnings when mistakes happen
- Reduced support burden
- Better error messages

**For Maintainers:**
- Single source of truth (`COMMON_FILTERS`)
- Auto-generated docs (never out of sync)
- Easy to add new filters (update dict → appears everywhere)
- Zero duplication across codebase

### Design Decisions

**Q: Why MCP Resource instead of verbose tool docstrings?**
- 55+ filters too verbose for tool signatures
- Resource fetched once, cached, referenced as needed
- Allows detailed examples without bloating context

**Q: Why auto-generation?**
- Eliminates drift between code and docs
- Add filter once → appears in tools + docs + validation
- Maintainability: single update point

**Q: Why warnings instead of errors?**
- Non-breaking for existing workflows
- Some parameters may be valid but undocumented
- Guidance without blocking

### Filter Validation Flow

```python
# Tool calls build_filters()
filters = build_filters(cve=cve_normalized, **kwargs)

# build_filters() validates
for param, value in kwargs.items():
    filter_name = COMMON_FILTERS.get(param)
    if not filter_name:
        # Track unknown parameters
        unknown_params.append(param)
    else:
        # Build filter
        filters.append({...})

# Log warnings for unknown parameters
if validate and unknown_params:
    logger.warning(f"Unknown filter parameters: {unknown_params}...")
```
